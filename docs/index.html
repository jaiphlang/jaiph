Samples

Jaiph workflows are typically executable .jh files with a shebang.

The core primitives are rule (read-only checks), prompt (calling agents), script (custom code or shell commands) and workflow (the main unit of orchestration). 

#!/usr/bin/env jaiph

script name_was_provided_check {
  # bash by default; use script:node, script:python3, etc. for other languages
  if [ -z "$1" ]; then
    echo "You didn't provide your name :(" >&2
    exit 1
  fi
}

rule name_was_provided {
  run name_was_provided_check("${arg1}")
}

workflow default {
  ensure name_was_provided("${arg1}")

  response = prompt "
    Say hello to ${arg1} and provide a fun fact about a person with the same name.
    Respond with a single line. Do not inspect files or run tools.
  "

  log "${response}"
}

Jaiph also supports structured output. You can use the following syntax: 

const response = prompt "..." returns '{ hello: string, fact: string }'

Then, Jaiph will add at runtime instructions to the prompt asking the agent to return a JSON object with the fields hello and fact, and it will parse the response and set the capture variable to the raw JSON string. The parser tolerates text before the JSON object (e.g. when the agent writes preamble before {...}). You can access the fields with dot notation: ${response.hello} and ${response.fact}. The underscore form (${response_hello}) also works but dot notation is preferred.

Jaiph has native support for testing. All files that end with *.test.jh are treated as tests.

And Jaiph highly recommends testing your workflows!

#!/usr/bin/env jaiph

import "say_hello.jh" as hello

test "without name, workflow fails with validation message" {
  # When
  response = hello.default allow_failure

  # Then
  expectEqual response "You didn't provide your name"
}

test "with name, returns greeting and logs response" {
  # Given
  mock prompt "Hello Alice! Fun fact: Alice in Wonderland was written by Lewis Carroll."

  # When
  hello.default "Alice"
}

This workflow uses Jaiph's ensure ... recover pattern to keep retrying a failing CI command after an agent-assisted fix. Inside a recover block, ${arg1} is the full merged stdout+stderr from the failed rule execution — including output from nested scripts and rules. The payload refreshes on each retry attempt. Persist ${arg1} to a stable path (here under .jaiph/tmp/) and assert it is non-empty before the prompt, so the message about "inspect the log file" matches what is on disk.

The workflow uses custom config to enable agent YOLO mode. Otherwise the agent cannot run some commands (for example npm test). By default Jaiph runs agents in their default, more secure modes.

#!/usr/bin/env jaiph

config {
  agent.backend = "cursor"
  agent.cursor_flags = "--force"
  # for Claude you can use:
  # agent.backend = "claude"
  # agent.claude_flags = "--permission-mode bypassPermissions"
}

script npm_run_test_ci {
  npm run test:ci
}

rule ci_passes {
  run npm_run_test_ci()
}

script save_string_to_file {
  printf '%s' "$1" > "$2"
}

workflow default {
  ensure ci_passes() recover {
    run save_string_to_file("${arg1}", ".jaiph/tmp/ci_failure.log")
    prompt "The CI check failed. Failure output is in .jaiph/tmp/ci_failure.log — fix the code so npm run test:ci passes."
  }
}

In the run below, ci_passes fails once. The failure output is captured in .jaiph/runs artifacts. Jaiph prompts the agent for a fix and retries until the check passes.

The inbox system lets workflows communicate through named channels. A workflow sends a message with <- (RHS: quoted literal, ${var}, or run to a script) and a route declaration (->) dispatches it to receivers with ${arg1}=message, ${arg2}=channel, ${arg3}=sender. 

#!/usr/bin/env jaiph

channel findings
channel report

workflow scanner {
  log "Scanning for issues..."
  findings <- "Found 3 issues in auth module"
}

workflow analyst {
  log "Analyzing message from ${arg3} on channel ${arg2}..."
  report <- "Summary: ${arg1}"
}

workflow reviewer {
  log "Reviewing message from ${arg3} on channel ${arg2}..."
  logerr "Critical issue: ${arg1}"
}

workflow default {
  findings -> analyst
  report -> reviewer

  run scanner()
}

This sample runs two prompt workflows in parallel: one with Cursor and one with Claude.

Each workflow sets its own agent.backend, captures the prompt response, and logs it. The default workflow uses run async to fan out both workflows concurrently, with an implicit join before completion.

#!/usr/bin/env jaiph

const prompt_text = 'Say: "Greetings! I am [model name]."'

workflow cursor_say_hello {
  config {
    agent.backend = "cursor"
  }
  const response = prompt "${prompt_text}"
  log "${response}"
}

workflow claude_say_hello {
  config {
    agent.backend = "claude"
  }
  const response = prompt "${prompt_text}"
  log "${response}"
}

workflow default {
  run async cursor_say_hello("${arg1}")
  run async claude_say_hello("${arg1}")
}

Running the workflow:

Core features

Running a workflow

./path/to/main.jh "arg1" "arg2" "arg3"

Arguments are passed positionally. In Jaiph strings (log, prompt, fail, send), use ${arg1}, ${arg2}. When you call a workflow, it executes the workflow default entrypoint.

By convention, keep Jaiph workflow files in <project_root>/.jaiph/ so workspace-root detection and agent setup stay predictable. Jaiph stores run artifacts in .jaiph/runs/ (step .out/.err logs plus run_summary.jsonl, an append-only JSONL event stream for reporting). Keep those out of git by adding .jaiph/runs/ to your .gitignore. See the CLI reference for event types, field requirements, and correlation rules.

Language

Combine prompts with strict checks. Compose rule, script, ensure, and prompt in one executable flow so checks and AI steps stay in the same pipeline. This way you can enforce structure over non-deterministic agent responses. See Grammar and Getting started (jaiph.org/getting-started).

Use Bash (or any language) in scripts. Command pipelines, checks, and background jobs belong in script blocks; workflows and rules call them with run. Use script:node, script:python3, script:ruby, etc. to select a common interpreter without writing a shebang line. For unlisted interpreters, add a manual #! shebang as the first body line. For trivial one-off commands, use inline scripts: run script("echo ok") — no named block needed.

Async calls. For async managed work, use run async wf() — Jaiph fans out the workflows concurrently and implicitly joins them before the parent workflow completes.

Agent inbox pattern (channels). Use inbox channels as a way to pass messages between workflows. Declare channels once at top level with channel <name>, then use channel_ref <- ... and channel_ref -> workflow (channel_ref can be local or imported, e.g. shared.findings). See Inbox & Dispatch.

Ensure recover pattern. ensure ... recover ... gives bounded self-healing retries: run a rule, apply a fix on failure, then retry up to the configured limit. See Grammar.

Runtime

Reporting server. Run jaiph report --workspace . to open a local read-only dashboard on .jaiph/runs: run history, step tree, log previews, raw .out/.err streaming, and an aggregate view, with live updates while workflows append run_summary.jsonl. See Reporting server.

Docker sandboxing. Enable isolated execution with Docker for stronger containment of agent and shell actions. Configure in config { runtime.* }. See Sandboxing.

Hooks. Attach shell automation to workflow and step lifecycle events via ~/.jaiph/hooks.json or <project>/.jaiph/hooks.json. See Hooks.

Configuration. Control behavior with config { ... } blocks at the module level or inside individual workflows for per-workflow overrides, plus environment variables (env wins precedence). See Configuration and CLI reference.

Testing Jaiph workflows. Test workflows with executable *.test.jh suites, mocks, and assertions to cover deterministic and agent-assisted paths. See Testing.

Syntax

Jaiph workflows

config { ... }
Optional runtime options (agent backend/flags, logs, runtime). Allowed at the top level (module-wide) and inside individual workflows (per-workflow overrides for agent.* and run.* keys). Environment variables override config values. See Configuration.
import "file.jh" as alias · const name = value / local name = value
Import other modules and define module-scoped variables (prefer const for new code) shared by rules, scripts, and workflows in the same file.
rule name { ... } · workflow name { ... } · script name { ... } · script:tag name { ... }
rule is for reusable checks (Jaiph structured steps; used with ensure), workflow orchestrates Jaiph steps only, and script holds bash (or any language via script:<tag> or a custom shebang) invoked with run. Scripts run in full isolation — only positional arguments and essential Jaiph variables (JAIPH_LIB, JAIPH_SCRIPTS, JAIPH_WORKSPACE) are inherited; module-scoped variables are not visible. Use source "$JAIPH_LIB/…" for shared utilities. Scripts are emitted as separate executable files under scripts/ (within the run build output tree; see CLI reference).
ensure ref() · ensure ref(args) recover { ... } · run ref() · run script("body")
ensure executes a rule (with optional args); recover adds bounded retry repair logic; run calls another workflow or module script from a workflow (managed step — same value/log contract as ensure). All calls require parentheses. run script("body") embeds a one-off shell command directly without a named script block — supports arguments, capture, and custom shebangs.
prompt "..." · const name = prompt "..." returns '{ field: type }'
Sends a prompt to the configured agent, optionally validates structured JSON output, and exposes captured variables.
const name = … · name = ensure ref() · name = run ref()
Bind or capture step values. For ensure / run to rules and workflows, explicit return "value" (or return run ref() / return ensure ref()) feeds the variable; for run to a script, capture follows stdout. const RHS has stricter rules (no $(...) — use run to a script). A call-like RHS must use the keyword explicitly — e.g. const x = run helper("${arg}"), not const x = helper("${arg}") (compile error with a correction hint). See Step output contract.
run async ref(args)
Run a workflow or script concurrently. All async steps are implicitly joined before the workflow completes; failures are aggregated. Workflows only — capture (const x = run async ...) is not supported. See Grammar.
fail "reason"
fail aborts with stderr + non-zero exit.
if [not] ensure ref() { } · if [not] run ref() { }
Brace conditionals only, with optional else if / else chains. Use run to a script to express command-based tests.
match $var { "lit" => …, /re/ => …, _ => … }
Pattern match on a string value. Arms are tested top-to-bottom; first match wins. Patterns: string literal (exact), regex, or _ wildcard. Usable as a statement, expression (const x = $var match { … }), or with return (return $var match { … }). Exactly one _ wildcard arm is required.
log "message" · logerr "message"
Emit informational or error output to the run tree while keeping workflow execution explicit and auditable. Terminal and tree text use echo -e-style backslash escapes; structured LOG / LOGERR payloads keep the raw message string.
channel <name> · channel_ref <- ... · channel_ref -> workflow
Declare channels at top level and then send/route messages between workflows. channel_ref supports local and imported refs (for example findings, shared.findings). Undefined refs fail validation. See Inbox & Dispatch.

Jaiph tests

test "description" { ... }
Defines a test case; files named *.test.jh execute as test suites. See Testing.
mock prompt "..." · mock prompt { /pattern/ => "response", _ => "default" }
Stub prompt calls with fixed responses or pattern-based dispatch so tests stay deterministic.
mock workflow · mock rule · mock script
Replace workflow, rule, or script implementations in tests for isolation and targeted assertions.
name = alias.workflow · allow_failure
Runs an imported workflow through the test harness (same managed semantics as run in .jh files): capture prefers an explicit return from the callee, otherwise combined output with internal event lines stripped. Add allow_failure when the test expects a non-zero exit. See Testing.
expectEqual · expectContain · expectNotContain
Core assertion helpers for exact matches and substring checks.
