What is Jaiph?

Jaiph is a composable scripting language and runtime for defining and orchestrating AI agent workflows. It supports native bash commands and a set of custom primitives to combine prompts with strict checks. 

curl -fsSL https://jaiph.org/install | bash

The installer will install the version 0.5.0 of Jaiph to ~/.local/bin. Check out the CHANGELOG to see what's new in the nightly branch and use jaiph use nightly to switch.

Samples

Jaiph workflows are typically executable .jh files with a shebang.

The core primitives are rule, prompt, script and workflow. Workflows and rules are Jaiph-only: ensure, run, const, brace if, fail, return, inbox <- / ->, … — not arbitrary bash lines. Bash belongs in script blocks (and shared .jaiph/lib loaded from there). Workflows call rules with ensure and call workflows or scripts with run. Pass arguments like bash scripts ($1, $2). 

#!/usr/bin/env jaiph

script arg_is_non_empty() {
  test -n "$1"
}

rule name_was_provided {
  if not run arg_is_non_empty "$1" {
    logerr "You didn't provide your name :("
    fail "You didn't provide your name"
  }
}

workflow default {
  ensure name_was_provided "$1"

  const response = prompt "
    Say hello to $1 and provide a fun fact about a person with the same name.
    Respond with a single line. Do not inspect files or run tools.
  "

  log "$response"
}

Jaiph also supports structured output. You can use the following syntax: 

const response = prompt "..." returns '{ hello: string, fact: string }'

Then, Jaiph will add at runtime instructions to the prompt asking the agent to return a JSON object with the fields hello and fact, and it will parse the response and set the capture variable to the raw JSON string. The parser tolerates text before the JSON object (e.g. when the agent writes preamble before {...}). You can access the fields with $response_hello and $response_fact.

Jaiph has native support for testing. All files that end with *.test.jh are treated as tests.

And Jaiph highly recommends testing your workflows!

#!/usr/bin/env jaiph

import "say_hello.jh" as hello

test "without name, workflow fails with validation message" {
  # When
  response = hello.default allow_failure

  # Then
  expectEqual response "You didn't provide your name"
}

test "with name, returns greeting and logs response" {
  # Given
  mock prompt "Hello Alice! Fun fact: Alice in Wonderland was written by Lewis Carroll."

  # When
  hello.default "Alice"
}

This workflow uses Jaiph's ensure ... recover pattern to keep retrying a failing CI command after an agent-assisted fix. Inside a recover block, $1 is the full merged stdout+stderr from the failed rule execution — including output from nested scripts and rules. The payload refreshes on each retry attempt. Persist $1 to a stable path (here under .jaiph/tmp/) and assert it is non-empty before the prompt, so the message about “inspect the log file” matches what is on disk.

The workflow uses custom config to enable agent YOLO mode. Otherwise the agent cannot run some commands (for example npm test). By default Jaiph runs agents in their default, more secure modes.

#!/usr/bin/env jaiph

config {
  agent.backend = "cursor"
  agent.cursor_flags = "--force"
  # for Claude you can use:
  # agent.backend = "claude"
  # agent.claude_flags = "--permission-mode bypassPermissions"
}

script npm_run_test_ci() {
  npm run test:ci
}

rule ci_passes {
  run npm_run_test_ci
}

script save_string_to_file() {
  printf '%s' "$1" > "$2"
}

workflow default {
  ensure ci_passes recover {
    run save_string_to_file "$1" ".jaiph/tmp/ci_failure.log"
    prompt "The CI check failed. Failure output is in .jaiph/tmp/ci_failure.log — fix the code so npm run test:ci passes."
  }
}

In the run below, ci_passes fails once. The failure output is captured in .jaiph/runs artifacts. Jaiph prompts the agent for a fix and retries until the check passes.

The inbox system lets workflows communicate through named channels. A workflow sends a message with <- (RHS: quoted literal, $var, or run to a script) and a route declaration (->) dispatches it to receivers with $1=message, $2=channel, $3=sender. 

#!/usr/bin/env jaiph

channel findings
channel report

workflow scanner {
  findings <- "Found 3 issues in auth module"
}

workflow analyst {
  report <- "Summary: $1"
}

script print_reviewed() {
  echo "[reviewed] $1"
}

workflow reviewer {
  run print_reviewed "$1"
}

workflow default {
  run scanner
  findings -> analyst
  report -> reviewer
}

This sample runs two prompt workflows in parallel: one with Cursor and one with Claude.

Each workflow sets its own agent.backend, captures the prompt response, and logs it. The default workflow starts both in background and waits for completion.

#!/usr/bin/env jaiph

const prompt_text = 'Say: "Greetings! I am [model name]."'

workflow cursor_say_hello {
  config {
    agent.backend = "cursor"
  }
  const response = prompt "$prompt_text"
  log "$response"
}

workflow claude_say_hello {
  config {
    agent.backend = "claude"
  }
  const response = prompt "$prompt_text"
  log "$response"
}

workflow default {
  run cursor_say_hello "$1" &
  run claude_say_hello "$1" &
  wait
}

Core features

Running a workflow

./path/to/main.jh "arg1" "arg2" "arg3"

Arguments are passed exactly like bash scripts ($1, $2, "$@"). When you call a workflow, it executes the workflow default entrypoint.

By convention, keep Jaiph workflow files in <project_root>/.jaiph/ so workspace-root detection and agent setup stay predictable. Jaiph stores run artifacts in .jaiph/runs/ (step .out/.err logs plus run_summary.jsonl, an append-only JSONL event stream for reporting). Keep those out of git by adding .jaiph/runs/ to your .gitignore. See the CLI reference for event types, field requirements, and correlation rules.

Language

Combine prompts with strict checks. Compose rule, script, ensure, and prompt in one executable flow so checks and AI steps stay in the same pipeline. This way you can enforce structure over non-deterministic agent responses. See Grammar and Getting started (jaiph.org/getting-started).

Use Bash (or any language) in scripts. Command pipelines, checks, and background jobs belong in script blocks; workflows and rules call them with run. Add a custom shebang (e.g. #!/usr/bin/env python3) to write scripts in other languages. For async managed work, use run wf & and a wait step.

Agent inbox pattern (channels). Use inbox channels as a way to pass messages between workflows. Declare channels once at top level with channel <name>, then use channel_ref <- ... and channel_ref -> workflow (channel_ref can be local or imported, e.g. shared.findings). See Inbox & Dispatch.

Ensure recover pattern. ensure ... recover ... gives bounded self-healing retries: run a rule, apply a fix on failure, then retry up to the configured limit. See Grammar.

Runtime

Reporting server. Run jaiph report --workspace . to open a local read-only dashboard on .jaiph/runs: run history, step tree, log previews, raw .out/.err streaming, and an aggregate view, with live updates while workflows append run_summary.jsonl. See Reporting server.

Docker sandboxing. Enable isolated execution with Docker for stronger containment of agent and shell actions. Configure in config { runtime.* }. See Sandboxing.

Hooks. Attach shell automation to workflow and step lifecycle events via ~/.jaiph/hooks.json or <project>/.jaiph/hooks.json. See Hooks.

Configuration. Control behavior with config { ... } blocks at the module level or inside individual workflows for per-workflow overrides, plus environment variables (env wins precedence). See Configuration and CLI reference.

Testing Jaiph workflows. Test workflows with executable *.test.jh suites, mocks, and assertions to cover deterministic and agent-assisted paths. See Testing.

Samples

Jaiph source code is built mostly with real Jaiph workflows. The .jaiph/docs_parity.jh workflow runs documentation maintenance checks, changelog updates, and cross-doc consistency guards. The .jaiph/engineer.jh workflow implements a queue-driven engineering loop that picks work, implements changes, verifies CI, and updates queue state. Shared bash helpers for those flows live under .jaiph/lib/ and load via source "$JAIPH_LIB/…" inside scripts. Generated workflow scripts export a default JAIPH_LIB rooted at ${JAIPH_WORKSPACE:-.}/.jaiph/lib so paths stay tied to the project workspace, not the build output directory.

Syntax

Jaiph workflows

config { ... }
Optional runtime options (agent backend/flags, logs, runtime). Allowed at the top level (module-wide) and inside individual workflows (per-workflow overrides for agent.* and run.* keys). Environment variables override config values. See Configuration.
import "file.jh" as alias · const name = value / local name = value
Import other modules and define module-scoped variables (prefer const for new code) shared by rules, scripts, and workflows in the same file.
rule name { ... } · workflow name { ... } · script name() { ... }
rule is for reusable checks (Jaiph structured steps; used with ensure), workflow orchestrates Jaiph steps only, and script holds bash (or any language via a custom shebang, e.g. #!/usr/bin/env node) invoked with run. Scripts run in full isolation — only positional arguments and essential Jaiph variables (JAIPH_LIB, JAIPH_SCRIPTS, JAIPH_WORKSPACE) are inherited; module-scoped variables are not visible. Use source "$JAIPH_LIB/…" for shared utilities. Scripts are emitted as separate executable files under build/scripts/.
ensure ref [args...] · ensure ref recover { ... } · run ref
ensure executes a rule (with optional args); recover adds bounded retry repair logic; run calls another workflow or module script from a workflow (managed step — same value/log contract as ensure).
prompt "..." · const name = prompt "..." returns '{ field: type }'
Sends a prompt to the configured agent, optionally validates structured JSON output, and exposes captured variables.
const name = … · name = ensure ref · name = run ref
Bind or capture step values. For ensure / run to rules and workflows, explicit return "value" feeds the variable; for run to a script, capture follows stdout. const RHS has stricter rules (no $(...) — use run to a script). A call-like RHS must use the keyword explicitly — e.g. const x = run helper "$arg", not const x = helper "$arg" (compile error with a correction hint). See Step output contract.
fail "reason" · wait
fail aborts with stderr + non-zero exit. wait joins async run … & jobs.
if [not] ensure ref { } · if [not] run ref { }
Brace conditionals only, with optional else if / else chains. Use run to a script to express command-based tests.
log "message" · logerr "message"
Emit informational or error output to the run tree while keeping workflow execution explicit and auditable. Terminal and tree text use echo -e-style backslash escapes; structured LOG / LOGERR payloads keep the raw message string.
channel <name> · channel_ref <- ... · channel_ref -> workflow
Declare channels at top level and then send/route messages between workflows. channel_ref supports local and imported refs (for example findings, shared.findings). Undefined refs fail validation. See Inbox & Dispatch.

Jaiph tests

test "description" { ... }
Defines a test case; files named *.test.jh execute as test suites. See Testing.
mock prompt "..." · mock prompt { if $1 contains "..." ; then respond "..." ; fi }
Stub prompt calls with fixed responses or content-based branching so tests stay deterministic.
mock workflow · mock rule · mock function
Replace workflow, rule, or script implementations in tests for isolation and targeted assertions (mock function is the test-file spelling).
name = alias.workflow · allow_failure
Runs an imported workflow through the test harness (same managed semantics as run in .jh files): capture prefers an explicit return from the callee, otherwise combined output with internal event lines stripped. Add allow_failure when the test expects a non-zero exit. See Testing.
expectEqual · expectContain · expectNotContain
Core assertion helpers for exact matches and substring checks.
