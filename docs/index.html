What is Jaiph?

Jaiph is a composable scripting language and runtime for defining and orchestrating AI agent workflows. It supports native bash commands and a set of custom primitives to combine prompts with strict checks. 

curl -fsSL https://jaiph.org/install | bash

The installer will install the version 0.3.0 of Jaiph to ~/.local/bin. Check out the CHANGELOG to see what's new in the nightly branch and use jaiph use nightly to switch.

Samples

Jaiph workflows are typically executable .jh files with a shebang.

The core primitives are rule, prompt, function and workflow. You can also embed any bash code inside your workflows and rules, and pass arguments to them in a similar way to bash scripts.

#!/usr/bin/env jaiph

rule name_was_provided {
  if [ -z "$1" ]; then
    echo "You didn't provide your name :(" >&2
    exit 1
  fi
}

workflow default {
  ensure name_was_provided "$1"

  response = prompt "
    Say hello to $1 and provide a fun fact about a person with the same name.
  "

  log "$response"
}

Running the workflow:

➜  ./say_hello.jh Jakub

running say_hello.jh

  ✓ 0s
  ▸ prompt “Say hello to Jakub and..." (Jakub)
  ✓ 9s

✓ PASS workflow default (8.9s)

Jaiph also supports structured output. You can use the following syntax: 

response = prompt "..." returns '{ hello: string, fact: string }'

Then, Jaiph will add at runtime instructions to the prompt asking the agent to return a JSON object with the fields hello and fact, and it will parse the response and set the capture variable to the raw JSON string. The parser tolerates text before the JSON object (e.g. when the agent writes preamble before {...}). You can access the fields with $response_hello and $response_fact.

Jaiph has native support for testing. All files that end with *.test.jh are treated as tests.

And Jaiph highly recommends testing your workflows!

#!/usr/bin/env jaiph

import "say_hello.jh" as hello

test "without name, workflow fails with validation message" {
  # When
  response=$({ hello.default 2>&1; } || true) # this way bash captures stderr

  # Then
  expectEqual response "You didn't provide your name"
}

test "with name, returns greeting and logs response" {
  # Given
  mock prompt "Hello Alice! Fun fact: Alice in Wonderland was written by Lewis Carroll."

  # When
  hello.default "Alice"
}

This is the workflow used in the Jaiph codebase to ensure compilation and tests pass. Note the conditionals, which are used in the same way as in bash, and the recursive flow: if compilation or tests fail, a prompt is triggered, and the ensure_ci_passes workflow runs again.

The workflow uses custom config to enable agent YOLO mode. Otherwise the agent cannot run some commands (for example npm test). By default Jaiph runs agents in their default, more secure modes.

#!/usr/bin/env jaiph

config {
  agent.backend = "cursor"
  agent.cursor_flags = "--force"
  # for Claude you can use:
  # agent.backend = "claude"
  # agent.claude_flags = "--permission-mode bypassPermissions"
}

rule project_compiles {
  npm run build
}

rule tests_pass {
  npm test
}

rule e2e_tests_pass {
  npm run test:e2e
}

workflow ensure_ci_passes {
  if ! ensure project_compiles; then
    prompt "Fix failing compilation so npm run build passes."
    run ensure_ci_passes
    return 0
  fi
  if ! ensure tests_pass; then
    prompt "Fix failing tests so npm test passes."
    run ensure_ci_passes
    return 0
  fi
  if ! ensure e2e_tests_pass; then
    prompt "Fix failing e2e tests so npm run test:e2e passes."
    run ensure_ci_passes
    return 0
  fi
}

workflow default {
  run ensure_ci_passes
}

Usage

Installation

curl -fsSL https://jaiph.org/install | bash

To install the nightly branch instead:

curl -fsSL https://jaiph.org/install | bash -s nightly

Then verify the installation: 

jaiph --version

Switch installed version:

jaiph use nightly
jaiph use 0.3.0

Running a workflow

./path/to/main.jh "arg1" "arg2" "arg3"

Arguments are passed exactly like bash scripts ($1, $2, "$@"). When you call a workflow, it executes the workflow default entrypoint.

By convention it is recommended to keep your workflows in the <project_root>/.jaiph/ directory. It ensures Jaiph detects workspace root and configures the agent correctly.

Jaiph stores all command output in .jaiph/runs/ directory. It is recommended to add it to your .gitignore.

Running tests

Jaiph supports native test scripts via the *.test.jh convention and first-class test "description" { ... } blocks. When you run the executable *.test.jh file, it is executed as a test.

Configuration

Runtime behavior is controlled by in-file config and environment variables. See configuration docs for precedence and keys.

For command syntax and flags, read CLI reference.

To run custom commands at workflow/step events, use hooks (~/.jaiph/hooks.json and <project>/.jaiph/hooks.json).

To get started from scratch: Getting started. For AI agents: Agent Skill.

Documentation

config { ... }
Optional block at the top of the entry file. Sets runtime options: agent.backend (cursor | claude), agent.command, agent.trusted_workspace, run.logs_dir, etc. Environment variables override. See Configuration.
import "file.jh" as alias
Imports rules and workflows from another Jaiph module under an alias. Jaiph compiles each module to a single shell file, and verifies the import at compile time.
rule name { ... }
Defines a reusable check/action that returns a shell exit code. The rule is executed in a read-only subshell, and its stdout is preserved. The rule fails if the command returns a non-zero exit code. Rules can consume positional parameters ($1, $2, "$@") forwarded by ensure. Inline brace-group short-circuits like cmd || { echo "failed"; exit 1; } are supported (single- or multi-line).
workflow name { ... }
Defines an orchestration entrypoint made of ordered steps. As opposed to rules, workflows can change the state of the system and are not restricted to read-only operations. Workflows fail if the script returns a non-zero exit code. Inline brace-group short-circuits like cmd || { echo "failed"; exit 1; } are supported (single- or multi-line).
function name() { ... }
Defines a reusable writable shell function. Functions can be called from workflows/rules and are tracked as regular Jaiph steps.
ensure ref [args...] · ensure ref recover <stmt> · ensure ref recover { stmt; ... }
Executes a rule, optionally forwarding arguments (e.g. ensure my_rule "$1"). With recover, the step becomes a bounded retry loop: run the rule; on failure run the recover body (one statement or a { stmt; stmt; ... } block); repeat until the rule passes or max retries is reached, then exit 1. Max retries default to 10; set JAIPH_ENSURE_MAX_RETRIES to override. See Grammar.
run ref
Executes another workflow from a workflow. run is not allowed inside a rule; use ensure to call another rule or move the call to a workflow.
Run tree and parameters
During jaiph run, the progress tree shows each step. In TTY mode, a single bottom line RUNNING workflow <name> (X.Xs) is updated in place (e.g. every second) and removed when the run finishes; the tree itself is identical to non-TTY (no per-line live timers). For workflow, prompt, and function steps called with arguments, the argument values are shown inline in gray after the step name (comma-separated in parentheses; no parameter names or internal refs). Workflow/function values are truncated at 32 characters. Prompt steps show a truncated preview of the prompt text (first 24 characters, then ... if longer) and the argument list is capped at 96 characters. See CLI reference.
log "message"
Displays a message in the progress tree at the current indentation depth. Takes a double-quoted string; shell variable interpolation ($var, ${var}) works at runtime. No spinner, no timing — a static annotation for marking workflow phases. See Grammar.
if ensure ref [args]; then ... [else ...] fi · if ! ensure ref [args]; then ... [else ...] fi · if ! <shell>; then ... fi
Conditional workflow steps. The positive form runs the then-branch when the rule succeeds; the negated form runs it when the rule fails. Both accept optional arguments after the rule reference and support an else branch. Then- and else-branches can contain run, prompt, and shell commands (but not ensure). if ! <shell>; then ... fi conditions on a shell expression.
prompt "..." · name = prompt "..." returns '{ field: type, ... }'
Sends prompt text to the configured agent (Cursor or Claude CLI; see agent.backend and Configuration). Without returns, stdout is sent to the agent and (if captured) stored in the variable. With returns '{ field: type, ... }', the prompt must capture to a variable; the agent is instructed to respond with one line of JSON; the runtime parses and validates it (flat schema; types string, number, boolean). On success, the capture variable holds the raw JSON and name_field is set for each schema field (e.g. $result_type). Invalid JSON, missing field, or type mismatch fails the step with distinct error codes. Logs and run_summary.jsonl are written to .jaiph/runs/.
Assignment capture: name = prompt "..." · name = prompt "..." returns '{ ... }' · name = ensure ref · name = run ref · name = <shell>
Capture a step's stdout into a variable. result = prompt "..." captures the agent's response. With returns '{ type: string, risk: string }', the response is validated as JSON and $result, $result_type, $result_risk, etc. are set. response = ensure tests_pass, out = run helper, and line = echo hello capture rule, workflow, or shell output. Exit behavior is unchanged: if the command fails, the step fails (use || true to capture on failure). stderr is not captured unless the command redirects it (e.g. 2>&1). See Grammar.
test "description" { ... }
Defines a test case. Files named *.test.jh are run as test suites.
Hooks (~/.jaiph/hooks.json, <project>/.jaiph/hooks.json)
Run custom shell commands at workflow/step lifecycle events (workflow_start, workflow_end, step_start, step_end). Project-local overrides global per event. Payload is JSON on stdin; failures are logged but do not block the run. See Hooks.
mock prompt "..."
In a test, stubs the next prompt call to return the given string instead of calling the agent.
mock prompt { if $1 contains "..." ; then respond "..." ; fi }
In a test, stub by prompt content: first matching branch wins; use elif / else as needed. See Testing.
mock workflow · mock rule · mock function
In a test, replace a workflow, rule, or function with a shell body for isolation. See Testing.
expectEqual var "value" · expectContain var "substring"
Test assertions: expectEqual checks exact match; expectContain checks that the variable contains the substring. Fail the test if the check does not hold.

All language primitives can be freely combined with bash code—they are not isolated or blocking, but instead are interoperable with standard shell scripting within your workflows and rules.
